# Bridging Two 技术说明

## 1. 系统目标

本工程包含两个 STM32U575 工程：

- `Bridging01`：板1，负责连接多个电脑侧串口，并通过 `USART3` 与板2通信。
- `Bridging02`：板2，负责接收板1发来的数据，处理后按原来源接口回传。

当前验证目标是：

1. 电脑可以从板1的多个串口入口发送数据。
2. 板1把不同入口的数据区分开，并统一通过 `USART3` 发给板2。
3. 板2处理数据后返回板1。
4. 板1根据来源编号，把回包只发回原来的串口。

目前板2的处理逻辑是：收到一包数据后，在数据末尾追加 `*` 再回传。

示例：

```text
电脑 -> 板1 USART1: 0000
板1 -> 板2 USART3: 来源=USART1, 内容=0000
板2 -> 板1 USART3: 来源=USART1, 内容=0000*
板1 -> 电脑 USART1: 0000*
```

## 2. 接口定义

### 2.1 板1接口

| 接口 | 波特率 | TX 引脚 | RX 引脚 | 用途 |
|---|---:|---|---|---|
| `USART1` | `115200` | `PA9` | `PA10` | 电脑侧入口 |
| `UART4` | `9600` | `PA0` | `PA1` | 电脑侧入口 |
| `LPUART1` | `115200` | `PC1` | `PC0` | 电脑侧入口 |
| `USART3` | `115200` | `PA7` | `PA5` | 板1与板2通信 |

### 2.2 板2接口

| 接口 | 波特率 | TX 引脚 | RX 引脚 | 用途 |
|---|---:|---|---|---|
| `USART3` | `115200` | `PA7` | `PA5` | 与板1通信 |

板2工程中其他串口虽然仍由 CubeMX 初始化，但当前业务逻辑只使用 `USART3`。

## 3. 接线方式

### 3.1 电脑连接板1

电脑 USB-TTL 可任选板1的一个电脑侧入口：

```text
USART1:
USB-TTL RX -> 板1 PA9
USB-TTL TX -> 板1 PA10

UART4:
USB-TTL RX -> 板1 PA0
USB-TTL TX -> 板1 PA1

LPUART1:
USB-TTL RX -> 板1 PC1
USB-TTL TX -> 板1 PC0
```

所有连接都必须共地：

```text
USB-TTL GND -> 板1 GND
```

### 3.2 板1连接板2

板1和板2之间使用 `USART3`：

```text
板1 PA7 (USART3_TX) -> 板2 PA5 (USART3_RX)
板1 PA5 (USART3_RX) -> 板2 PA7 (USART3_TX)
板1 GND             -> 板2 GND
```

## 4. 数据流

整体数据路径如下：

```text
电脑侧 USART1  ┐
电脑侧 UART4   ├──> 板1按端口独立收包 ──USART3──> 板2
电脑侧 LPUART1 ┘                         <──USART3──
                 板1按来源编号发回原接口
```

板1的三个电脑侧串口不是简单混合成一个裸数据流，而是先各自独立成帧，再带来源编号发送给板2。

## 5. 为什么需要来源编号

如果三个电脑侧串口都周期发送，例如：

```text
USART1:  每 1000ms 发送一次
UART4:   每 800ms 发送一次
LPUART1: 每 400ms 发送一次
```

那么多个串口的数据可能在很接近的时间进入板1。如果直接把字节合并到同一条 `USART3` 裸流中，板2可能看到：

```text
000011112222
```

这时板2无法判断：

- 哪些字节来自 `USART1`
- 哪些字节来自 `UART4`
- 哪些字节来自 `LPUART1`
- 回包应该发回哪个串口

因此当前方案使用“带来源编号的数据帧”。

## 6. 板间协议

板1和板2之间的 `USART3` 不传裸数据，而传结构化帧。

### 6.1 帧格式

```text
+------+-------+---------+--------+-----------+----------+
| SOF0 | SOF1  | Port ID | Length | Payload   | Checksum |
+------+-------+---------+--------+-----------+----------+
| 0x55 | 0xAA  | 1 byte  | 1 byte | 0~128字节 | 1 byte   |
+------+-------+---------+--------+-----------+----------+
```

字段说明：

| 字段 | 说明 |
|---|---|
| `SOF0` | 固定帧头第 1 字节，值为 `0x55` |
| `SOF1` | 固定帧头第 2 字节，值为 `0xAA` |
| `Port ID` | 来源串口编号 |
| `Length` | 有效数据长度 |
| `Payload` | 实际数据 |
| `Checksum` | 校验和 |

### 6.2 端口编号

| Port ID | 板1串口 |
|---:|---|
| `1` | `USART1` |
| `2` | `UART4` |
| `3` | `LPUART1` |

### 6.3 校验和

校验和计算方式：

```text
Checksum = Port ID + Length + Payload所有字节
```

计算结果按 `uint8_t` 保存，也就是自动保留低 8 位。

## 7. 板1工作方式

板1主要完成四件事：

1. 轮询 `USART1 / UART4 / LPUART1` 三个电脑侧串口。
2. 每个串口独立收集数据。
3. 某个串口超过 `30ms` 没有新字节后，认为这一包结束。
4. 把这一包打包成带 `Port ID` 的帧，通过 `USART3` 发给板2。

板1收到板2回包后：

1. 从 `USART3` 解析出完整帧。
2. 读取帧中的 `Port ID`。
3. 根据 `Port ID` 找到原电脑侧串口。
4. 只把数据发回原来的串口。

也就是说：

```text
USART1 发出的数据，只回 USART1
UART4 发出的数据，只回 UART4
LPUART1 发出的数据，只回 LPUART1
```

## 8. 30ms 空闲成帧

“30ms 空闲成帧”的意思是：

某个串口收到数据后，板1先暂存起来。如果该串口在最后一个字节到来后连续 `30ms` 没有收到新字节，板1就认为这一包数据结束。

例如电脑发送：

```text
0000
```

实际串口上是逐字节到达：

```text
0 -> 0 -> 0 -> 0
```

板1收到最后一个 `0` 后，如果等待 `30ms` 没有新字节，就把这 4 个字节作为一包：

```text
Payload = 0000
```

这个方法适合当前串口助手周期发送测试，因为 `1000ms / 800ms / 400ms` 都远大于 `30ms`。

### 8.1 边界情况

如果同一个串口在 `30ms` 内连续发送两包，例如：

```text
0000
1111
```

并且两包之间间隔小于 `30ms`，板1可能把它们合并为一包：

```text
00001111
```

如果后续需要严格区分每一包，应改为：

- 固定结束符成帧，例如每包以 `\r\n` 结束。
- 长度字段协议，例如电脑侧也发送 `长度 + 数据`。
- STM32 `IDLE + DMA` 接收方案。

## 9. 为什么当前方案能解决三串口周期发送问题

当前方案的关键是：

1. 三个电脑侧串口分别独立缓存。
2. 每个串口独立判断 30ms 空闲。
3. 板间传输时带 `Port ID`。
4. 板1回包时按 `Port ID` 路由。

因此三个口即使同时自动发送，也不会互相污染来源信息。

示例：

```text
USART1 发送 0000 -> Port ID = 1 -> 回 USART1
UART4  发送 1111 -> Port ID = 2 -> 回 UART4
LPUART1发送 2222 -> Port ID = 3 -> 回 LPUART1
```

板间 `USART3` 上可能连续出现多个帧，但每个帧都有自己的来源编号，所以板1和板2都能正确区分。

## 10. 当前实现方式

当前代码使用：

- 主循环轮询寄存器 `ISR / RDR / TDR`
- 禁用对应串口的接收中断
- 软件方式判断 `30ms` 空闲
- 自定义帧头和校验和

暂未使用：

- DMA
- UART IDLE 中断
- RTOS 消息队列

这种实现的优点：

- 逻辑直观
- 调试方便
- 适合当前数据量和周期发送测试

需要注意：

- 单包最大载荷当前为 `128` 字节。
- 同一个串口两包间隔小于 `30ms` 时可能被合并。
- 当前发送函数是阻塞式等待 `TXE`，数据量很大时会占用主循环时间。

## 11. 后续可升级方向

### 11.1 IDLE + DMA

后续可以把电脑侧串口和板间串口升级为：

```text
UART DMA circular buffer + IDLE interrupt
```

优点：

- 不需要软件等待 `30ms`
- 数据停止后可快速触发帧完成
- CPU 占用更低
- 更适合高波特率和大数据量

成本：

- 需要配置多个 UART DMA
- 需要处理 DMA 写指针、IDLE 中断、溢出和缓存切换
- 代码复杂度更高

### 11.2 电脑侧显式协议

如果上位机可控，也可以让电脑侧每包数据带结束符：

```text
0000\r\n
```

或者带长度：

```text
LEN=4, DATA=0000
```

这样板1不需要依赖 `30ms` 空闲判断，包边界会更明确。

## 12. 测试建议

1. 重新编译并烧录 `Bridging01` 到板1。
2. 重新编译并烧录 `Bridging02` 到板2。
3. 板1与板2用 `USART3` 交叉连接并共地。
4. 电脑分别连接板1的 `USART1 / UART4 / LPUART1` 测试。
5. 串口助手设置为对应波特率：
   - `USART1`: `115200`
   - `UART4`: `9600`
   - `LPUART1`: `115200`
6. 发送 `0000`，预期当前接口收到：

```text
0000*
```

三个串口同时周期发送时，预期每个串口只收到自己发送内容对应的回包。
